Java设计模式之单例模式

什么是单例模式

所谓单例模式，就是用一定的方法，让某个类在整个软件系统运行过程中，只包含一个对象实例，并且只提供一个获取该类实例对象的方法(static method), 比如一些重量级的对象，例如Hibernate中的SessionFactory，就是一个重量级对象，应该保证其单例，这时候就用到了单例模式

单例模式的八种实现方式

  1. 饿汉式（static variable）
  2. 饿汉式（static block）
  3. 懒汉式（线程不安全）
  4. 懒汉式（线程安全，synchronized method）
  5. 懒汉式（线程安全，synchronized block）
  6. 双端检索机制（double check）
  7. 静态内部类
  8. 枚举

单例模式遵循如下几个步骤即可

分为如下几个步骤

  • 构造函数私有化（防止通过new 创建对象）
  • 在类的内部创建对象
  • 向外暴露这个对象（写一个公有的方法，返回instance对象）

饿汉式（static variable）的优缺点

  • 优点： 此方法简单，在类加载的时候，就已经完成了该实例的创建，避免了线程的同步问题
  • 缺点： 在类加载的时候就完成了加载，没有实现懒加载的效果，如果此项目从始至终没有使用过这个单例对象实例，会造成内存的浪费
  • 结论：这个方法可以用，但是可能造成内存浪费

饿汉式（static block）

饿汉式（static block） 的优缺点

  • 这种方法，和第一种方法，一样，都是在类加载的时候初始化单例对象，只不过把创建实例的代码放到了static代码块里，本质上没有区别，优缺点与第一个相同
  • 结论：这个方法可以用，但是可能造成内存浪费

懒汉式（线程不安全）

懒汉式（线程不安全） 的优缺点

  • 优点：起到了Lazy loading 的效果，但是此方法，只能在单线程的环境下使用。
  • 如果在多线程的环境下，线程A进入到instance == null的判断中，判断成功，但是还没进入到创建的逻辑，此时，线程A被挂起，线程B也进行了instance == null的判断，也通过了，此时，就会产生多个Singleton对象，所以，在多线程环境下，并不可以用这种方式。
  • 结论：在实际开发中，不要使用这种方式。

懒汉式（线程安全，synchronized method）

懒汉式（线程安全，synchronized method）

  • 优点：解决了线程安全的问题
  • 效率过低，当每个线程需要获取这个单例对象的时候，都需要进行同步，但是其实，只需要执行一次的实例化代码(new Singleton())即可，后续其他线程，都是执行return逻辑，所以这种方法会导致效率低下
  • 结论：在实际开发中，不要使用这种方式。

懒汉式（线程安全，synchronized block）

懒汉式（线程安全，synchronized block）

  • 优点：相比于同步方法的方式来讲，只锁住了创建实例的代码块，其余逻辑，都是直接返回，效率更高
  • 缺点：但是此方式下，还是会产生多线程的问题，因为创建对象[new Singleton()]这个步骤，在底层分为了好几步(分配内存空间，初始化对象，将刚分配的内存地址赋值给instance对象)，
  • 结论：在实际开发中，不要使用这种方式。
