Spring IOC GetBean流程分析

从 AnnotationConfigApplicationContext#getBean() 开始

调用到了父类 AbstractApplicationContext 的getBean

父类 getBean调用BeanFactory 的getBean() 方法

因为 在GenericApplicationContext 中，默认创建的BeanFactory类型为DefaultListableBeanFactory，而DefaultListableBeanFactory 继承自 AbstractBeanFactory，所以最终调用到的是 AbstractBeanFactory#getBean()

getBean调用doGetBean，doGetBean 最最主要的逻辑

transformedBeanName(name) 将beanName转化为bean的真实名称（有可能传入的是bean的别名，所以需要转化）

getSingleton(beanName) 尝试从单例缓存池中获取

单例缓存池中获取到了

单例缓存池中没有获取到，执行创建bean的逻辑

判断当前获取的bean是不是原型的并且在循环依赖中

否则，获取其父容器，判断当前的容器有没有父容器

ps. 为什么这里有父子容器的概念， 在SpringMVC中，进行创建的时候，就会创建出一对父子容器，在父容器存储Service，Repository， 在子容器存储 Controller，所以在SpringMVC中，就可以通过Controller找到Service，找到Repository，而不能通过 Service找到Controller

然后检查一下当前的bean是否为抽象bean

然后判断当前bean的创建是否需要依赖于其他的bean，例如 A 的创建依赖 B

如果 当前bean是单例bean，走创建单例bean的创建流程

getSingleton(beanName, () -> {};

createBean(beanName, mbd, args);

解析出当前的beanDefinition 中的Class对象

Object bean = resolveBeforeInstantiation(beanName, mbdToUse); 创建切面，AOP的关键

doCreateBean(beanName, mbdToUse, args)

createBeanInstance(beanName, mbd, args);

return beanInstance 返回回去，getBean结束
