设计模式

一、设计模式的目的

编写软件的过程中，面临着耦合性，内聚性，可维护性，可扩展性，重用性，灵活性等，方面的挑战，设计模式即为了让程序具有更好的

  1. 代码的重用性
  2. 可读性
  3. 可扩展性
  4. 可靠性
  5. 呈现高内聚，低耦合的特性

二、设计模式的七大原则

设计模式原则是在编码时，应当遵守的原则，即设计模式为什么要这么设计的依据

  1. 单一职责原则
  2. 接口隔离
  3. 依赖倒置
  4. 里氏替换原则
  5. 开闭原则
  6. 迪米特法则
  7. 合成复用原则

2.1 单一职责原则

2.1.1 基本介绍

​ 对于类来说，即一个类只负责一项职责，如A类负责了两个不同的职责，职责1，职责2，当职责1需求变更而修改A类时，可能导致职责2错误，所以应当将A类按照职责分为A1，A2，例如一个Dao即负责User的crud，又负责Order的crud，则应该将这个Dao分为UserDao和OrderDao

2.1.2 案例

1. 违反单一职责原则,
2. 遵守单一职责原则，将不同的职责分到不同的类中
3.未拆分类，在方法级别上遵守单一职责原则，只有在类足够简单才能在方法上遵守单一职责原则

2.1.3 单一职责原则的注意事项

  1. 降低了类的复杂性，一个类只负责一个职责
  2. 提高了类的可读性，可维护性
  3. 降低了修改代码引起的风险
  4. 通常情况下，需要遵守单一职责，除非逻辑足够简单，可以在代码级违反单一职责原则，只有在类中的方法足够少，才能在方法级别上保持单一职责。

2.2 接口隔离原则

2.2.1 基本介绍

​ 1. 客户端不应该依赖它不需要的接口，即一个类对另一个类的依赖应该建立在最小接口上

​ 2. A通过Interface1依赖B，C通过Interface1依赖D，如果Interface1对于A和C不是最小接口，那么B和D必须去实现不需要的方法

​ 3. 依据接口隔离原则应该将Interface1拆分成几个接口，类A，C分别于需要的接口进行依赖

2.3 依赖倒置原则(依赖倒转)

2.3.1 基本介绍

​ 依赖倒转原则是指：

  1. 高层模块不应该依赖底层模块，应该都为抽象

  2. 抽象不应该依赖于细节，细节应该依赖于抽象

  3. 依赖倒转的中心思想是面向接口编程

  4. 依赖倒转是基于这样的理念，相对于细节的多变，抽象的东西更稳定，以抽象为基础搭建的架构比以细节为基础的架构更加稳定，在java中，抽象指接口或抽象类，细节为具体的实现类。

  5. 使用接口或抽象类的目的是制定好规范，不涉及具体的操作，把细节的任务交给子类去实现

2.3.2 应用实例

不符合依赖倒置原则demo: 当我们需要接口更多类型的消息时，需要在Person中添加相应的方法来接口
改进，抽象出一个接口消息的接口，需要接口消息的类实现相应的接口即可

2.3.3 依赖倒置的三种方式

  1. 接口传递
  2. 构造方法传递
  3. setter方法传递

2.3.4 注意事项

  1. 底层模块尽量都有抽象类和接口，或者都有，程序稳定性更好
  2. 变量的声明类型尽量是抽象类或接口，利于程序的扩展
  3. 继承时遵守里氏替换原则

2.4 里氏替换原则

​ 2.4.1 基本介绍

  1. 如果对每个类型为T1的对象o1，都有类型为T2的对象o2，使得以T1定义的所有程序P在所有对象o1都能替换成o2时，程序P的行为没有发生变化，那么类型T2是类型T1的子类型，换句话说：所有引用基类的地方必须能透明的使用其子类的对象。
  2. 在使用继承时，遵守里氏替换原则，在子类中尽量不要重写父类的方法
  3. 继承让两个类的耦合性增强了，在适当的情况下，可以通过聚合，组合，依赖来解决
